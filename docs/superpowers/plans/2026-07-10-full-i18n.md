# Full i18n — internationalize the ENTIRE application

**Date:** 2026-07-10
**Status:** in progress — see §Progress at the end
**Scope:** every user-visible string in the product — SPA, mailers, push, DB-persisted copy,
AI output, PDF reports, client portal, landing/legal pages, error pages, PWA manifest,
ActiveAdmin, Stripe/MP descriptions — plus locale-aware dates, numbers, money, and plurals.

## 1. Architecture decisions

### D1 — Locales
`pt-BR` (default, source language) + `en` as the first target. Structure everything so adding
`es` later is "add one file per namespace". Locale codes: BCP-47 (`pt-BR`, `en`).

### D2 — Locale resolution model (the core design)

There are **three distinct audiences**, each with its own locale:

| Audience | Locale source | Applies to |
|---|---|---|
| **App user** | `users.locale` (new column; default from workspace at signup) | SPA UI, personal emails (auth, digest, assignment), push notifications, ActiveAdmin (staff) |
| **Agency's client** | `clients.locale` (new column; default from workspace) | Client portal, approval portal + emails, invoice emails/checkout, report PDF, WhatsApp copy |
| **Anonymous visitor** | `?locale` param → cookie → `Accept-Language` → `pt-BR` | Landing, legal, login/signup, error pages, public connect |

Plus one **content language** knob, separate from UI locale:
- `clients.content_language` (default `pt-BR`) — the language of the *audience of the client's
  social content*. AI-generated content (carousel copy, captions, scripts, video storyboards,
  scope, summaries) follows **content_language**, not the UI locale. A US user managing a
  Brazilian client still gets PT captions; the reverse too.

Resolution plumbing:
- Backend: `Current.locale` set in `ApplicationController` via `around_action` →
  `I18n.with_locale(resolve_locale)`. API requests: `Current.user.locale`. Public/token
  endpoints: the token's client/workspace locale. Jobs/mailers: explicit
  `I18n.with_locale(recipient.locale)` at every send/render site (never rely on ambient locale
  in Sidekiq).
- Frontend: locale delivered in the auth/bootstrap payload (`/me`), stored by the i18n provider;
  anonymous pages read cookie/navigator.

### D3 — Frontend library
**react-i18next + i18next** (`i18next-icu` for ICU plurals/selects). De facto standard, React 19
compatible, supports namespaces, lazy-loading, and interpolation. JSON resource files live in
`app/frontend/locales/<locale>/<namespace>.json`, bundled by Vite (static import for pt-BR,
lazy chunk for others).

Namespaces mirror the page/domain dirs: `common`, `board`, `ticket`, `calendar`, `clients`,
`projects`, `posts`, `studio`, `billing`, `settings`, `auth`, `portal`, `approval`, `reports`,
`meetings`, `dashboard`, `tasks`, `account`, `errors`, `toasts`, `validation`.

### D4 — Backend framework
Rails I18n + the `rails-i18n` gem (ships complete pt-BR/en for AR validations, dates, numbers).
`config.i18n.default_locale = :'pt-BR'`, `available_locales = [:'pt-BR', :en]`,
`fallbacks = [:'pt-BR']`. Keys organized per surface: `config/locales/{mailers,api,operations,
push,notes,credits,reports,pages,admin}/{pt-BR,en}.yml`.

### D5 — DB-persisted copy: store keys, not prose
System notes, push notifications, and credit-transaction descriptions are currently written as
rendered PT text. Rendered-at-write can never be re-localized. Change to **key + params,
rendered at read/send time**:
- `notes` (system kind): add `i18n_key` (string) + `i18n_params` (jsonb). Serializer renders
  `I18n.t(key, **params, locale: Current.locale)` when `i18n_key` present; legacy rows fall back
  to the stored `body`. Human-authored notes stay as-is (user content is never translated).
- Push: `Operations::Push::Notify` accepts `title_key`/`body_key` + params; renders per
  recipient in `I18n.with_locale(user.locale)` at send time.
- `credit_transactions`: add `description_key` + `description_params`; serializer renders at
  read time, falls back to legacy `description`.
- Rule going forward: **no rendered copy is ever persisted** — always key + params.
  User-generated content (ticket titles, briefs, notes, captions) is data, never translated.

### D6 — AI prompts
`Prompts::Base` gains a `response_language` (resolved from `Current.workspace`/client
`content_language` for content prompts, from `Current.user.locale` for UI-ish prompts like
ticket summaries — decide per prompt class; default: content prompts → `content_language`,
summaries/retrospective/audit → workspace locale). Replace every hardcoded "responda em
português" with an interpolated language directive. `AiConfig` untouched.

### D7 — Routes: keep PT segments canonical, centralize paths now
Localizing URL segments (`/quadro` vs `/board`) doubles the router table, breaks bookmarks, and
complicates the SPA catch-all — **not worth it in v1**. Decision: PT paths stay canonical for all
locales (they're brand-neutral enough), but **extract all ~118 path literals into
`lib/paths.js`** (`paths.ticket(id)`, `paths.board`, …) so localized routes become a later
drop-in if ever wanted. Marketing pages (SSR, SEO-relevant) DO get locale variants:
`/en`, `/en/pricing`, … with `hreflang` alternates.

### D8 — Formatting
Rewrite `lib/formatters.js` to read the active locale from the i18n instance:
- `brl()` → `money(cents, currency = 'BRL')` using `Intl.NumberFormat(locale, {currency})`.
  **Currency stays BRL** everywhere money moves (Stripe plans, MP invoices, credit pricing) —
  only display formatting localizes. Keep `brl` as an alias during migration.
- `timeAgo`/`relativeDay` → `Intl.RelativeTimeFormat`.
- Dates/numbers: pass locale through; calendar week start becomes locale-dependent
  (`pt-BR`: Monday — keep; `en`: Monday too is acceptable, configurable constant).
- BR masks (CPF/CNPJ/CEP/phone) are **domain formats, not locale** — they stay, because clients
  billed via Mercado Pago are Brazilian regardless of UI language. Label them via i18n
  ("CPF/CNPJ" stays as the term).

### D9 — Enforcement (keeps it done)
- ESLint rule (`no-literal-strings` scoped to JSX text/attrs like `title`, `placeholder`,
  `label`, `aria-label`) — error on new hardcoded literals outside `locales/`.
- Backend: a spec that greps `app/{services,mailers,controllers,models}` for accented chars /
  a PT wordlist outside `config/locales` and fails with the offending lines.
- `i18n-tasks` gem (backend) + `i18next-parser` (frontend) in CI: missing/unused key detection,
  and en↔pt-BR parity check.
- Update `CLAUDE.md` language rules: "user-facing strings live ONLY in locale files; hardcoded
  PT (or EN) copy in code is a violation."

## 3. Phases

Ordering principle: infrastructure first, then the centralized maps (80% of visible surface for
20% of effort), then the long-tail sweeps, then client-facing/backends, then static/marketing.
Every phase ships green with pt-BR pixel-identical to today.

### Phase 1 — Foundation (backend + frontend infra) 【~1 dia】
1. Gems: `rails-i18n`, `i18n-tasks`. Config: default/available locales, fallbacks.
2. Migrations above. `Registration` sets `user.locale` from workspace; Settings/Account/Client
   forms get locale selects (Phase 7 for UI polish; plumbing now).
3. `Current.locale` + `ApplicationController` `around_action { I18n.with_locale(...) }`;
   public/token controllers resolve from the token's client/workspace.
4. npm: `i18next`, `react-i18next`, `i18next-icu`. `app/frontend/i18n.js` (init, namespaces,
   pt-BR inline / en lazy), provider in `App.jsx`, locale from the `/me` bootstrap payload +
   `<html lang>` sync.
5. `lib/paths.js` — extract all route-path literals; codemod the ~118 references in ~27 files.
6. CI guards (D9): ESLint rule (warn-level until Phase 3 completes, then error),
   backend PT-grep spec (scoped per-directory, expanding as phases land), key-parity checks.

### Phase 2 — Frontend: centralized maps + formatters 【~1 dia, huge visible coverage】
1. `lib/constants.js`: every `label/short/hint/title/placeholder/description` value → `t()` keys
   (`common:status.ideation.label`, …). Meta objects keep structure; values become keys resolved
   through a helper (`labelFor(STATUS_META, key)`) or getters bound to i18next.
2. `charts/labels.js` `SERIES_LABELS`, `calendarUtils.js` weekday/month names →
   `Intl.DateTimeFormat(locale)`; kill the direct `toLocaleDateString('pt-BR')` calls
   (also `TimeGrid.jsx:218`).
3. `formatters.js` rewrite per D8 (locale-driven, `Intl.RelativeTimeFormat`, `money()`).
4. `POSITIONING_STEPS` + `positioningFields.jsx` → `clients` namespace.
5. `creativeName.js`, `roles.js`.

### Phase 3 — Frontend: full sweep (~130 files) 【the long tail, 3–5 dias; parallelizable by domain】
Sweep one namespace/domain at a time (each its own commit, review, and locale-file pair):
1. `toasts`: all 112 toast call sites (`hooks/data/*`, `useAuth`, `useTicket`, `useBoard`,
   `useStrategy`, `api/client.js` error surface) → `toasts` namespace. Note: hooks are outside
   React components — use the i18next instance directly (`i18n.t`), not `useTranslation`.
2. Domain order (density-first): `ticket/` components + Tickets pages → `Settings` → `Billing`
   → `Clients` → `Studio` → `Calendar`/`Meetings` → `Dashboard`/`Reports`/`Posts` → `Projects`
   → `Tasks`/`Account` → `Auth`/`Errors` → layout/nav/`ui` primitives (`EmptyState` defaults,
   confirm dialogs, filter bars).
3. Portal + Approval (`pages/Portal`, `pages/Approval`, `components/portal/*`,
   `components/approval/*`) — these read `clients.locale` delivered in the token-scoped payload,
   NOT the user locale.
4. Hand-rolled plurals (~14 sites) → ICU messages. Inline validation messages → `validation` ns.
5. Add document-title management while at it (small `usePageTitle(t('ns:title'))` hook) — titles
   become translatable for free.
6. Flip the ESLint rule to `error`.

### Phase 4 — Backend: API messages, operations, persisted copy 【2–3 dias】
1. Controller-services (~34 files): every raised/rendered PT message → `I18n.t('api.…')`.
   Errors raised for the SPA get stable keys; the frontend keeps displaying server-sent text
   (already localized server-side by `Current.locale`).
2. Push call sites (~10+): switch to `title_key`/`body_key` + params; `Push::Notify` renders per
   recipient locale (D5).
3. System notes (~9 sites in `approvals/*`, `autopilot/*`, `ai/build_scope`,
   `tickets/cascade_fields`, `tickets/spawn_follow_up`) → `i18n_key`/`i18n_params`; note
   serializer renders at read time; legacy rows fall back to `body`.
4. Credit-transaction descriptions (6 ops + `billing/sync_subscription`) → key+params, rendered
   in serializers and ActiveAdmin.
5. `Pricing` feature bullets, `MEDIA_LABELS` (`tickets/publish.rb`), Stripe checkout/product
   descriptions (`create_credit_checkout_session`, `provision_plan_prices` — workspace locale),
   remaining PT in models.

### Phase 5 — Mailers + PDF 【1–2 dias】
1. All 17 mailers: subjects → `default_i18n_subject`/`t('.subject')`; wrap every delivery in
   `I18n.with_locale(recipient locale)` — recipient = user for auth/digest/ticket mailers,
   client for approval/invoice/report mailers.
2. 58 templates (+ layout + text parts) → `t('.key')` lazy lookup. Keep the branded layout;
   footer/legal boilerplate → `mailers.layout.*`.
3. Report PDF: `reports/pdf.html.erb` + `layouts/report_pdf.html.erb` → `t()`, `lang` attr from
   client locale; `Public::Portal::ReportPdf` renders in client locale.

### Phase 6 — AI prompts + content language 【1 dia】
1. `Prompts::Base#response_language`; replace pt-BR directives in the 13 classes.
2. Content prompts (`CarouselCopy`, `FieldFill`, `ScopeBuilder`, `VideoStoryboard`,
   `VideoEditor`, `VideoPromptImprover`, `StrategyPlanner`, `ClientPositioning`,
   `ClientFromLandingPage`) → client `content_language`. Workspace-facing prompts
   (`TicketSummary`, `Retrospective`, `ProjectAudit`) → workspace locale.
3. Client form (frontend + `Operations::Clients::*`) exposes `content_language`.

### Phase 7 — Public surface: landing, legal, static, PWA, switcher UI 【1–2 dias】
1. Marketing pages (7 views + `marketing.html.erb`): extract to `pages.*` keys; add `/en` route
   scope with localized slugs, `hreflang` alternates, `og:locale` per locale, meta descriptions.
   Legal pages: EN versions are legal documents — translate deliberately, not mechanically.
2. SPA shell (`spa/index.html.erb`): `lang` + `<title>` from resolved locale.
3. Static error pages (`public/*.html`, 7 files): minimal bilingual pages (they're served
   without app context) — PT headline + EN subline, or locale-suffixed variants if the proxy
   can negotiate.
4. PWA manifest description → served per locale (it's already ERB).
5. `public_connect/*`, `data_deletion/show` → `t()` with visitor-locale resolution.
6. Locale switcher UI: `/conta` (user locale), Settings → workspace default locale,
   client form → client locale + content language. Changing user locale hot-swaps i18next +
   invalidates queries (server re-renders API messages/notes in new locale).
7. ActiveAdmin: `active_admin.{pt-BR,en}.yml` for the 14 resources' menu/labels (staff-facing;
   lowest priority — do last).
8. Seeds (`db/seeds.rb` pricing copy): plan bullets move to `Pricing` keys (done in Phase 4);
   seeds keep only English identifiers.

### Phase 8 — QA + translation pass 【1 dia】
1. `i18n-tasks health` + `i18next-parser` clean: zero missing, zero unused, en/pt-BR parity.
2. Grep-spec green over the whole `app/` tree (guard scope = everything).
3. Manual QA in `en`: board, ticket lifecycle end-to-end, generation flow, billing/paywall,
   portal + approval via token, invoice email + PDF, push, digest.
4. Native-quality review of the `en` files (the codebase's PT is the source of truth; EN must
   not read machine-translated).
5. Update `CLAUDE.md` (language rules section) + `docs/ARCHITECTURE.md` with the locale model.

## 4. Explicit non-goals (v1)

- Localized SPA URL segments (D7 — path helper makes it possible later).
- Multi-currency (money is BRL; only formatting localizes).
- Translating user-generated content (briefs, captions, notes, ticket titles).
- RTL support.
- Regionalizing BR-domain masks/flows (CPF/CNPJ, Pix, Mercado Pago are product decisions, not i18n).

## 5. Risks & mitigations

- **Volume regression risk** (~350 files): phase-per-domain commits, pt-BR output must stay
  byte-identical — snapshot the extracted pt-BR values straight from today's literals.
- **Sidekiq ambient locale leaks**: never set `I18n.locale=`; only `with_locale` blocks. Add a
  spec asserting jobs don't depend on ambient locale.
- **Read-time note rendering perf**: `I18n.t` is cheap; params are pre-resolved at write time
  (names interpolated as data, not re-queried).
- **Key drift** between en/pt-BR: CI parity check (Phase 1.6) from day one.
- **Prompt language changes AI output shape**: JSON tool-forced outputs unaffected (keys are
  English); only prose language flips. Verify `FieldFill`/`ScopeBuilder` goldens in en.
